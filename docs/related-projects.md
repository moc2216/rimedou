# 相关项目调研

> 本文为早期调研记录。本项目触发键现已改为**右 Command**（右 Ctrl 仅作豆包“免按模式”语音键、由工具合成），结束语音改为任意键，弹窗靠 dwell 解决（见 `spec.md` / `design.md` 的“当前实现”）。文中涉及“本项目用右 Ctrl 触发”的表述以此为准；对两个参考仓库本身的分析仍然有效。

## 结论

两个参考仓库都证明了一个共同方向：豆包语音可以被“切到豆包输入源 + 触发豆包语音快捷键 + 恢复原输入源”这种方式外接出来。

但它们不能直接照搬到本项目。原因是本项目的核心约束更具体：

- 用户主输入法固定优先恢复到鼠须管简体：`im.rime.inputmethod.Squirrel.Hans`。
- 豆包输入法内的“免按模式”快捷键是事实来源，当前为右 Ctrl。
- 触发键（右 Cmd）还要让渡给 Type4Me 等外部语音工具。
- 用户需要显眼的菜单栏启动、暂停、恢复鼠须管和退出能力。
- 不默认安装 Hammerspoon、Karabiner、LaunchAgent 或其他全局依赖/系统配置。

最值得借鉴的是 `Doubao-Voice-Input-Bridge` 的“焦点回弹”判断：豆包输入源切换成功，不等于当前文本输入上下文已经重新绑定到豆包语音模块。这个判断和本项目试用中出现的“前几次成功，后面无反馈或停留豆包输入法”高度相关。

## 参考仓库

- `Paxxs/Doubao-ime-hammerspoon`：https://github.com/Paxxs/Doubao-ime-hammerspoon
- `xubihang/Doubao-Voice-Input-Bridge`：https://github.com/xubihang/Doubao-Voice-Input-Bridge

## `Doubao-ime-hammerspoon` 分析

这个项目是 Hammerspoon 配置方案。

核心流程：

- 监听右 Command。
- 记录当前输入法。
- 切到豆包输入法。
- 模拟双击左 Option。
- 延迟约 2 秒恢复原输入法。

可借鉴点：

- 把“原输入源”记录为一次会话内状态，而不是长期写入配置。
- 触发后延迟恢复，避免语音结果还没提交就切走。
- watcher 出错时用保护层包住回调，避免监听器静默失效。
- 用全局变量持有 Hammerspoon watcher，避免被 GC 或 reload 边缘情况影响。

不适合照搬：

- 它依赖 Hammerspoon，并且安装脚本会安装全局工具、写入 `~/.hammerspoon/init.lua`，不符合本项目“不默认安装全局依赖或修改系统配置”的约束。
- 默认触发键是右 Command，不是用户当前豆包设置中的右 Ctrl。
- 触发豆包语音使用双击左 Option，不匹配用户当前只开启的右 Ctrl 免按模式。
- 它用输入法名称 `豆包输入法` 匹配，本项目已验证并使用输入源 ID，更稳定。
- 它没有 Type4Me 让渡逻辑，也没有菜单栏暂停/恢复鼠须管。

## `Doubao-Voice-Input-Bridge` 分析

这个项目是 Swift 原生菜单栏 app，结构与本项目更接近。

核心流程：

- 监听全局触发快捷键，默认右 Command。
- 按住超过阈值后开始语音流程。
- 记录当前输入源。
- 切到豆包输入法并等待确认。
- 做一次 app 焦点回弹：短暂切到中性 app，再切回原 app/window。
- 发送豆包语音快捷键，默认左 Option。
- 松开触发键后释放语音快捷键，并恢复原输入源。

它的关键判断：

- 只切输入源并发送语音快捷键时，第二次唤醒可能失败。
- 可工作的修复是模拟“焦点离开再回来”，让 macOS 的文本输入上下文重新绑定。
- 问题不完全在键盘事件，而更可能在 Doubao 的 InputMethodKit / AppKit 文本客户端绑定。

可借鉴点：

- `focusBounce`：切到豆包后、触发语音前，让当前文本输入上下文重新绑定。
- 延迟配置化：`postSwitchSettleDelay`、`focusBounceBackDelay`、`focusBounceSettleDelay`、`restoreDelay` 等都适合做成可调参数。
- 更细的状态机：区分等待触发、准备语音、语音热键已按住、恢复输入法。
- 菜单栏里提供打开配置、检查权限、打开日志等运维入口。
- 权限窗口比只依赖系统弹窗更容易理解。
- 稳定签名可以减少反复授权的麻烦。

不适合直接照搬：

- 它是“按住说话”模型，本项目当前是匹配豆包右 Ctrl 免按模式的一次按下开始、再次按下结束。
- 默认语音快捷键是左 Option，不是用户当前豆包设置的右 Ctrl。
- 它的触发键和语音键不同；如果把二者都改为右 Ctrl，可能出现自触发、循环监听或与豆包弹窗冲突，需要重新设计。
- 它默认会安装用户级 LaunchAgent 并 KeepAlive 自动重启。本项目当前阶段更需要用户可见、可手动退出；自动守护属于后续明确需求，不能默认启用。
- 它恢复的是本次触发前输入源；本项目根据用户需求优先恢复鼠须管，避免豆包停留为日常输入法。

## 与本项目现状对比

本项目当前已经具备：

- Swift 原生菜单栏 app。
- 触发键（右 Cmd）监听 + 任意键结束语音。
- 豆包输入源 ID 与鼠须管输入源 ID 验证。
- Type4Me 运行时自动让渡。
- 暂停、启用、恢复鼠须管、退出菜单。
- 退出和暂停时尽力恢复鼠须管。
- 空闲状态下每秒守护，发现停留豆包时拉回鼠须管。

本项目当前不足：

- 语音启动流程没有焦点回弹，因此可能仍遇到“切到豆包成功，但豆包语音模块没有绑定当前输入框”的不稳定问题。
- 延迟参数仍写死在代码里，不便于根据试用反馈微调。
- 状态机只有 `idle / suspended / doubaoVoiceActive / error`，对“切换中、等待上屏、恢复中”等过程表达不够细。
- 日志还偏终端输出，不适合正式试用时定位偶发问题。
- 当前 ad-hoc 签名会导致每次替换应用后重新授权，正式使用前需要考虑稳定签名。

## 下一步建议

优先级 1：不要改成 Hammerspoon 方案。

理由：它解决的是相似问题，但引入外部自动化依赖，不符合本项目约束，也不匹配右 Ctrl 免按模式。

优先级 2：保留当前右 Ctrl 免按模型。

理由：豆包设置中已经明确是右 Ctrl，本项目应该跟随豆包设置，而不是要求用户改豆包快捷键。

优先级 3：把焦点回弹作为可配置实验能力加入本项目。

建议新增配置：

```json
{
  "enableFocusBounce": true,
  "focusBounceBackDelaySeconds": 0.16,
  "focusBounceSettleDelaySeconds": 0.16,
  "postDoubaoSwitchDelaySeconds": 0.18,
  "primaryRestoreDelaySeconds": 1.2
}
```

建议流程：

1. 用户按右 Ctrl。
2. 工具确认 Type4Me 未运行。
3. 切到豆包输入源。
4. 确认当前输入源为豆包。
5. 如果 `enableFocusBounce=true`，短暂激活本工具或 Finder，再切回原 app/window。
6. 再合成一次右 Ctrl，让豆包按它自己的免按模式开始语音。
7. 用户第二次按右 Ctrl 后，不额外合成停止键，等待上屏窗口提交，再恢复鼠须管。
8. 空闲守护继续兜底，把偶发停留豆包的状态拉回鼠须管。

优先级 4：增加更可靠的右 Ctrl 物理状态判断。

当前代码通过 keyCode 区分左右 Ctrl，但通过通用 `.maskControl` 判断 Ctrl 是否按下。参考仓库使用左右修饰键的 raw flag mask。后续可改为按右 Ctrl 专属 flag 判断物理按下状态，降低极端组合键场景下的状态错判。

优先级 5：暂不默认启用 LaunchAgent。

理由：用户已经要求显眼的启动和关闭方式。自动守护会写入 `~/Library/LaunchAgents` 并改变退出语义，属于后续明确确认后再做的能力。

## 建议实施顺序

1. 先写 spec：新增“焦点回弹可配置实验能力”的验收行为。
2. 再写设计：明确为什么它只在切到豆包后、合成右 Ctrl 前执行。
3. 再写计划：拆分 `FocusBouncer`、配置字段、状态机测试、手动验收。
4. TDD 实现配置解析和状态机行为。
5. 实现 AppKit / Accessibility 的焦点回弹服务。
6. 构建 app，复制到 `/Applications`，重新授权后手动测试 5 到 10 次短按。

## 决策记录

当前决策：

- 不照搬任一仓库。
- 保留本项目 Swift 原生路线。
- 保留右 Ctrl 免按模式匹配豆包设置。
- 借鉴 `Doubao-Voice-Input-Bridge` 的焦点回弹思路。
- 暂不引入 Hammerspoon、LaunchAgent 自动守护、按住说话模型。

后果：

- 下一轮改动会比单纯调延迟稍大，需要先更新 spec / design / plan。
- 焦点回弹会让第一次唤醒略增加约 0.3 秒，但可能换来更稳定的连续唤醒。
- 如果焦点回弹影响当前应用焦点体验，需要保留配置开关，允许关闭。
