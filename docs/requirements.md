# 需求澄清

## 结论

本项目要解决的核心问题不是“做一个完整输入法管理器”，而是做一个极小的 macOS 辅助工具：

- 平时保持鼠须管作为主要输入法。
- 用户按右 Ctrl 时，进入豆包语音输入。
- 用户再次按右 Ctrl 结束语音输入后，自动恢复到鼠须管。
- 豆包只作为语音输入入口，不作为日常键盘输入法。

## 触发键演进（重要）

本文档为初版需求澄清，原文以**右 Ctrl 作为用户触发键**。实际实现已演进（见 `spec.md` / `design.md` 的“当前实现”）：

- 用户触发键改为**右 Command**（`triggerKey`，可配置）；右 Ctrl 仅作为豆包“免按模式”语音键，由工具合成。
- 结束语音改为**任意键**（不再限于右 Ctrl）。
- “语音唤起方式调整”弹窗靠合成右 Ctrl 的 dwell 解决。

下文凡“按右 Ctrl 触发/结束”均以此演进为准。

## 已确认信息

- 运行环境是 macOS，设备是 Mac mini。
- 当前存在两个输入法：
  - `/Library/Input Methods/DoubaoIme.app`：豆包输入法。
  - `/Library/Input Methods/Squirrel.app`：鼠须管（Rime）。
- 用户日常使用鼠须管输入 86 五笔。
- 用户想用豆包语音输入，但不想使用豆包其他输入方式。
- 期望交互键是右 Ctrl。
- 用户还有另一个语音输入工具：`/Applications/Type4Me.app`。
- 当另一个语音输入工具运行时，本工具应自动停用豆包语音逻辑，不接管右 Ctrl。
- 另一个语音输入工具路径应做成配置项，方便后续替换。
- 按右 Ctrl 唤醒豆包语音输入。
- 语音输入结束后，再按右 Ctrl。
- 第二次按右 Ctrl 后，输入焦点的输入状态应自动变回鼠须管。

## 默认需求判断

### 目标行为

工具启动后应在后台运行，监听右 Ctrl 这个按键。

当配置的外部语音输入工具正在运行时：

1. 本工具不响应右 Ctrl。
2. 本工具不切换到豆包。
3. 本工具不尝试结束豆包语音输入。
4. 用户可以用外部语音输入工具自己的右 Ctrl 逻辑。

当配置的外部语音输入工具未运行时，本工具才接管右 Ctrl。

当用户当前处于普通输入状态时：

1. 用户按右 Ctrl。
2. 工具切换到豆包输入法。
3. 工具触发或允许豆包进入语音输入状态。
4. 用户开始语音输入。

当用户正在使用豆包语音输入时：

1. 用户再次按右 Ctrl。
2. 工具优先尝试结束豆包语音输入。
3. 工具把当前输入法恢复到鼠须管。
4. 用户继续用鼠须管打字。

如果豆包语音输入无法被外部可靠停止，第一版至少必须做到恢复鼠须管。

### 非目标

第一版不做以下事情：

- 不管理所有输入法。
- 不做菜单栏完整偏好设置。
- 不同步配置到多台设备。
- 不替代鼠须管或豆包输入法。
- 不修改豆包或鼠须管的内部文件。
- 不默认修改系统级输入法配置。

### 用户体验默认值

- 工具应尽量无感运行。
- 右 Ctrl 的行为应稳定、可预期。
- 当配置的外部语音输入工具正在运行时，右 Ctrl 归外部语音输入工具使用。
- 当配置的外部语音输入工具退出后，右 Ctrl 才归本工具使用。
- 如果切换失败，应尽量给出可见反馈或日志，而不是静默失败。
- 启动工具后，不应破坏用户已有输入法顺序和输入习惯。
- 工具退出后，不应留下异常输入状态；至少应尽力恢复鼠须管。

## 风险和不确定点

### 右 Ctrl 是否能被可靠区分

macOS 可以区分左右修饰键，但具体实现需要验证：

- 是否能只监听右 Ctrl，不影响左 Ctrl。
- 是否能在不同应用中全局监听。
- 是否需要辅助功能权限或输入监听权限。
- 是否会和系统快捷键、应用快捷键冲突。

### 豆包语音输入能否被外部可靠唤醒

需要验证豆包输入法的语音输入入口：

- 切换到豆包输入法后，右 Ctrl 是否就是豆包默认的语音开关键。
- 豆包是否要求自己的快捷键事件必须由真实键盘触发。
- 外部工具是否能用系统 API、AppleScript、快捷键模拟或其他方式触发。
- 语音输入状态是否可查询，还是只能由本工具维护一个“估计状态”。

### 输入法标识需要本机确认

需要确认鼠须管和豆包在 macOS 输入法系统里的真实标识，例如：

- bundle id。
- input source id。
- localized name。

不能只根据 `.app` 路径硬编码。

### 权限模型

第一版很可能需要下列权限之一：

- 辅助功能权限。
- 输入监听权限。
- 自动化权限。

是否需要这些权限要在设计阶段验证，不能提前假设。

用户接受第一版需要手动授予必要的 macOS 权限。项目要求是只申请必要权限，不偷偷修改系统配置，不做超出输入法切换和快捷键处理以外的事情。

## 待确认问题

### 必须确认

已确认：

1. 右 Ctrl 可以用于语音输入用途。除豆包语音和配置的外部语音输入工具外，用户没有其他右 Ctrl 用途。
2. 当配置的外部语音输入工具正在运行时，本工具暂停接管右 Ctrl。
3. 第二次右 Ctrl 的目标是结束豆包语音并恢复鼠须管；如果豆包无法外部停止，至少必须恢复鼠须管。
4. 如果豆包语音不能被外部直接唤醒，接受“切到豆包后模拟豆包现有语音快捷键”的替代方案。
5. 第一版接受需要手动授予 macOS 辅助功能或输入监听权限。

### 可以稍后确认

1. 工具是否需要开机自启动。
2. 是否需要菜单栏图标。
3. 是否需要显示当前状态，例如“鼠须管 / 豆包语音中”。
4. 是否需要配置文件，还是第一版直接写死鼠须管和豆包。
5. 是否需要日志文件，或只在开发阶段输出日志。
6. 配置项是否只包含外部语音输入工具路径，还是也包含输入源标识和快捷键策略。

## 第一版成功标准

第一版只要满足以下标准即可认为可用：

1. 用户启动工具。
2. 用户在任意普通输入框中使用鼠须管输入。
3. `/Applications/Type4Me.app` 或配置的外部语音输入工具未运行。
4. 用户按右 Ctrl。
5. 豆包语音输入被唤起或进入可语音输入状态。
6. 用户语音输入完成后再次按右 Ctrl。
7. 当前输入法恢复为鼠须管。
8. 用户可以立即继续用鼠须管输入 86 五笔。

外部语音输入工具正在运行时：

1. 用户启动工具。
2. `/Applications/Type4Me.app` 或配置的外部语音输入工具正在运行。
3. 用户按右 Ctrl。
4. 本工具不切换豆包，不恢复鼠须管，不干扰右 Ctrl。
5. 用户可以正常使用外部语音输入工具。

## 第一版失败处理标准

出现失败时，第一版至少需要能判断或提示以下情况：

- 配置的外部语音输入工具路径无效。
- 未找到鼠须管输入源。
- 未找到豆包输入源。
- 缺少必要权限。
- 切换输入法失败。
- 右 Ctrl 监听失败。
- 豆包语音输入无法被外部触发。
