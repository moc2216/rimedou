# 功能规格

## 结论

一个 macOS 后台输入法辅助工具：外部语音工具未运行时，按右 Command（可配置触发键）唤起豆包语音输入；语音中按任意键结束，工具恢复鼠须管。

## 当前实现（以本节为准）

> 下文 v0.1 原始章节以“右 Ctrl 作触发键”描述初版设计，**触发与结束相关行为已被本节及文末《触发键与语音键分离规格》《任意键结束语音规格》取代**。右 Ctrl 现仅作为豆包“免按模式”语音键（由工具合成）。

- **触发键 = 右 Command**（`triggerKey`，可配置，默认 `rightCommand`）。用户按它进入/结束语音。
- **语音键 = 右 Ctrl**：豆包“免按模式”快捷键，工具切到豆包后**合成 1 次**右 Ctrl 启动语音；与触发键分离，避免豆包双击判定。
- **任意键结束**：语音激活期间按任意常规键（空格/回车/字母等），豆包原生停止语音，工具切回鼠须管。
- **弹窗治理**：合成右 Ctrl 时 keyDown 与 keyUp 之间加 0.1 秒停留（dwell），避免“零停留瞬按”被判快速双击而弹“语音唤起方式调整”。
- **让渡**：Type4Me（外部语音工具）运行时，工具不接管触发键。
- **只开一个实例**：终端 `--run` 与菜单栏 app 不可同时运行，否则双实例各自合成右 Ctrl，制造弹窗与误切。

## 术语

- 主输入法：鼠须管（Rime / Squirrel）。
- 豆包输入法：`/Library/Input Methods/DoubaoIme.app` 对应的输入法。
- 外部语音工具：默认是 `/Applications/Type4Me.app`，路径应可配置。
- 右 Ctrl：键盘右侧 Control 修饰键。
- 豆包语音快捷键：豆包输入法设置中“语音输入 -> 免按模式”的快捷键。当前用户设定为“右 Ctrl”。
- 接管右 Ctrl：本工具把右 Ctrl 作为豆包语音开关处理。第一版采用旁听策略，不吞掉物理右 Ctrl。
- 暂停接管：本工具不处理右 Ctrl，让其他工具使用。

## 配置规格

第一版至少包含以下配置项：

- `externalVoiceAppPath`：外部语音工具路径，默认值为 `/Applications/Type4Me.app`。
- `externalVoiceBundleId`：外部语音工具 bundle id，默认值为 `com.type4me.app`。
- `primaryInputSourceId`：主输入法输入源，默认值为鼠须管简体 `im.rime.inputmethod.Squirrel.Hans`。
- `doubaoInputSourceId`：豆包输入源，默认值为 `com.bytedance.inputmethod.doubaoime.pinyin`。
- `doubaoVoiceHotkey`：豆包语音快捷键，第一版只支持 `rightControl`，必须与豆包“免按模式”的设置一致。

配置项用于判断外部语音工具是否正在运行、切换输入源，并匹配豆包语音快捷键。

第一版可以暂时不提供图形配置界面，但配置位置和修改方式必须在设计阶段明确。

## 状态规格

工具运行时至少区分以下状态：

- `idle`：普通状态，本工具可以接管右 Ctrl。
- `suspended`：外部语音工具正在运行，本工具暂停接管右 Ctrl。
- `doubaoVoiceActive`：本工具已尝试进入豆包语音输入流程。
- `error`：出现无法继续的错误，例如缺少权限或找不到输入源。

状态切换规则：

- 当外部语音工具正在运行时，工具应进入 `suspended`。
- 当外部语音工具退出后，工具应离开 `suspended`，恢复到可接管右 Ctrl 的状态。
- 当用户在 `idle` 下按右 Ctrl，工具进入 `doubaoVoiceActive`。
- 当用户在 `doubaoVoiceActive` 下再次按右 Ctrl，工具应恢复主输入法，并回到 `idle`。

## 外部语音工具让渡规格

当 `externalVoiceAppPath` 对应的应用正在运行时：

1. 用户按右 Ctrl。
2. 本工具不得切换到豆包输入法。
3. 本工具不得恢复鼠须管输入法。
4. 本工具不得模拟任何用于豆包语音的快捷键。
5. 本工具应保持暂停接管状态，直到外部语音工具退出。

验收标准：

- Type4Me 正在运行时，右 Ctrl 的行为不被本工具改变。
- Type4Me 退出后，右 Ctrl 可重新用于豆包语音流程。

## 进入豆包语音规格

当外部语音工具未运行，且工具处于 `idle` 时：

1. 用户按右 Ctrl。
2. 工具识别这是 `doubaoVoiceHotkey`，而不是左 Ctrl。
3. 工具延迟处理这次事件，让物理右 Ctrl 先被当前鼠须管状态消化。
4. 工具切换当前输入法到豆包输入法。
5. 工具等待系统实际报告当前输入源已经是豆包。
6. 工具按豆包设置中的同一快捷键触发豆包语音输入。
7. 工具进入 `doubaoVoiceActive`，保持豆包语音流程继续运行。

验收标准：

- 左 Ctrl 不触发豆包语音流程。
- 右 Ctrl 触发豆包语音流程。
- 豆包只应收到一次有效的右 Ctrl 触发，不应出现重复触发导致的“语音唤起方式调整”。
- 触发后，用户可以开始使用豆包语音输入。

## 退出提交规格

当用户第二次按右 Ctrl 退出豆包语音流程：

1. 工具等待豆包把识别结果提交到输入框。
2. 工具恢复鼠须管。
3. 工具不得再次向豆包发送右 Ctrl 停止快捷键。

验收标准：

- 正常短按流程不受影响。
- 第二次右 Ctrl 后，已识别文字应优先上屏，不应被切换输入法或 `Esc` 打断。

## 退出豆包语音规格

当工具处于 `doubaoVoiceActive` 时：

1. 用户再次按右 Ctrl。
2. 工具尽快切换当前输入法到鼠须管。
3. 工具回到 `idle`。

验收标准：

- 第二次右 Ctrl 后，当前输入法必须恢复为鼠须管。
- 第二次右 Ctrl 不再额外向豆包发送停止快捷键，也不自动发送 `Esc`，避免取消豆包尚未提交的识别结果。

## 空闲守护规格

当工具处于 `idle`，且外部语音工具未运行时：

1. 工具应定期检查当前输入法。
2. 如果当前输入法不是鼠须管，工具应自动切回鼠须管。
3. 如果 Type4Me 等外部语音工具正在运行，工具不得强行恢复鼠须管。

验收标准：

- 语音流程结束后即使一次恢复失败，工具也应在空闲轮询中把输入法拉回鼠须管。
- 工具不应干扰外部语音工具运行时的右 Ctrl 行为。

## 输入源识别规格

工具不得只根据 `.app` 路径硬编码输入源。

工具需要在运行或初始化阶段确认：

- 鼠须管输入源存在。
- 豆包输入源存在。
- 鼠须管输入源可被系统选择。
- 豆包输入源可被系统选择。
- 能够把输入法切换到鼠须管。
- 能够把输入法切换到豆包。

验收标准：

- 找不到鼠须管时，工具进入错误状态并提示原因。
- 找不到豆包时，工具进入错误状态并提示原因。
- 输入源已安装但未启用时，工具进入错误状态并提示需要用户手动启用。
- 切换失败时，工具进入错误状态或输出明确错误信息。

## 权限规格

如果工具需要 macOS 权限，应只请求必要权限。

可能需要的权限包括：

- 辅助功能权限。
- 输入监听权限。
- 自动化权限。

验收标准：

- 权限不足时，工具能明确提示缺少什么权限。
- 权限不足时，工具不得静默伪装成正常运行。
- 工具不得修改系统设置来绕过权限。

## 启动和退出规格

启动时：

1. 加载配置。
2. 检查外部语音工具运行状态。
3. 检查必要输入源。
4. 检查必要权限。
5. 开始监听触发键（右 Cmd）。

退出时：

1. 停止监听触发键。
2. 如果工具认为自己处于 `doubaoVoiceActive`，应尽力恢复鼠须管。
3. 不修改输入法列表，不修改系统输入法配置。

## 菜单栏控制规格

正式试用版应提供菜单栏图标，避免用户必须记忆终端命令。

菜单栏至少包含：

- 当前状态：运行中、已暂停、Type4Me 运行中已让渡、豆包语音中或错误。
- `暂停` / `启用`：暂停或恢复触发键接管。
- `恢复鼠须管`：立即尝试把当前输入法切回鼠须管。
- `退出`：关闭后台进程。

验收标准：

- 用户可以通过菜单栏退出程序。
- 用户可以通过菜单栏暂停程序，暂停后不再监听触发键。
- 用户可以通过菜单栏重新启用程序。
- 用户可以通过菜单栏恢复鼠须管。

## 日志和反馈规格

第一版至少需要开发者可见的反馈，例如标准输出日志。

日志至少覆盖：

- 配置加载结果。
- 外部语音工具运行状态变化。
- 输入源识别结果。
- 权限检查结果。
- 触发键（右 Cmd）触发进入豆包语音。
- 任意键（或触发键）结束语音并恢复鼠须管。
- 切换失败或触发失败。

## 非功能规格

- 工具应能在后台运行。
- 工具不应明显影响普通键盘输入。
- 工具不应依赖网络。
- 工具不应修改豆包或鼠须管应用内部文件。
- 工具不应安装全局依赖。
- 工具不应默认写入系统级配置目录。

## 暂不纳入第一版

- 开机自启动。
- 图形配置界面。
- 多个外部语音工具同时配置。
- 完整输入法管理。
- 自动安装或配置 macOS 权限。

## 触发键与语音键分离规格

### 背景
试用反馈：豆包"语音唤起方式调整"弹窗频繁出现。经本机验证确认根因——豆包在检测到"右 Ctrl 被快速连按两下"时弹窗（原生双击也会弹，慢按不弹）；本工具原机制"用户真按右 Ctrl + 工具假按右 Ctrl"恰好构成快速双击，导致弹窗概率大增。

### 配置规格
- 新增 `triggerKey`：用户按下的触发键，默认 `rightCommand`；缺省该字段时按默认值处理。
- 保留 `doubaoVoiceHotkey`：工具合成给豆包的语音键，仍为 `rightControl`，与豆包"免按模式"设置一致，不改。
- 二者分离后，豆包只收到工具合成的 1 个右 Ctrl，不再构成双击。

### 触发规格
- `idle` 下按 `triggerKey`（默认右 Cmd）一次：工具切到豆包，合成 1 次右 Ctrl 触发语音，进入 `doubaoVoiceActive`。
- `doubaoVoiceActive` 下再按 `triggerKey` 一次：工具恢复鼠须管，回到 `idle`；退出时不合成右 Ctrl（与现状一致）。
- 用户按右 Ctrl（豆包自己的语音键）时：本工具不响应，完全交给豆包原生处理。

### 让渡规格（不变）
- Type4Me 运行时仍让渡 `triggerKey`。

### 验收标准
- 右 Cmd 单按：进入豆包语音。
- 右 Cmd 再按：恢复鼠须管。
- 右 Ctrl 按下：工具不做任何事，不影响豆包原生语音。
- 连续多次触发语音后，不再出现“语音唤起方式调整”弹窗（已验：靠《合成按键停留修复》实现，**非**触发键分离；触发键分离本身不消除弹窗）。
- 左 Cmd 及其他修饰键：不触发。
- `triggerKey` 可配置，缺省 `rightCommand`。

### 非本版范围
- 焦点回弹暂不做（弹窗根因是双击，非焦点）。
- 触发键暂只支持 `rightCommand`，结构上预留扩展。

## 任意键结束语音规格

### 背景
豆包输入法原生支持“任意键停止语音”：语音输入过程中，按下空格、回车或其他任意常规按键，豆包都会自行停止语音。用户希望 RimeDou 借用这一点——说完话按任意键即可结束语音并切回鼠须管。

### 结束规格
- 工具处于 `doubaoVoiceActive` 时，任意常规按键（`keyDown`，如空格、回车、字母）应使工具切回鼠须管并回到 `idle`。
- 豆包自身会在该按键时停止语音，工具不再合成任何停止键。
- 事件监听为 `listen-only`，结束键必然透传：按键送达应用时豆包仍是当前输入法，由豆包自行处理（与原生体验一致），随后工具再切回鼠须管。
- 工具处于 `idle` 等非激活状态时，任意按键不做任何事。
- 本版“任意键”仅指常规 `keyDown`；除触发键外的纯修饰键（flagsChanged）不作为结束键。

### 验收标准
- 右 Cmd 进入语音后，按空格/回车等任意键：语音停止，输入法恢复鼠须管。
- 结束键不应在输入框多打出多余字符（真机验收点）。
- 空闲时任意按键不影响工具状态，也不产生切换动作。
- `suspended`（Type4Me 运行）下任意按键不触发任何动作。
