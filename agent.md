# 项目规则

## 项目目标

本项目用于开发一个 macOS 输入法切换工具：

- 日常输入默认使用鼠须管（Rime / Squirrel）输入 86 五笔。
- 偶尔按右 Ctrl 唤醒豆包输入法的语音输入。
- 再次按右 Ctrl 结束语音输入后，自动恢复输入状态到鼠须管。
- 目标是只借用豆包语音输入能力，不把豆包作为日常键盘输入法。

## 工作原则

- 先写规则，再写代码。
- 先澄清需求，再写 spec。
- 先做设计和计划，再实现。
- 默认采用 TDD：先写能表达目标行为的测试，再写实现。
- 每个阶段完成后等待确认，再进入下一阶段。
- 技术文档优先使用中文。
- 代码、命令、变量名使用英文。

## 红线操作

以下操作必须先获得明确确认：

- 删除文件、目录或 git 历史。
- 修改 `.env`、密钥、token、CI/CD 配置。
- 数据库 schema 变更或数据迁移。
- `git push`、`git rebase`、`git reset --hard`、强制推送。
- 安装新的全局依赖或修改系统配置。
- 公开发布，例如 npm publish、生产部署、发文章。

## 目录结构约定

项目根目录保留以下顶层结构：

- `agent.md`：项目协作规则，本文件。
- `docs/`：需求、spec、设计、决策记录等中文文档。
- `src/`：项目源码。
- `tests/`：自动化测试。
- `scripts/`：本项目内使用的辅助脚本。
- `fixtures/`：测试夹具或示例数据。

目录创建规则：

- 新目录创建前，先明确它的职责。
- 不创建用途模糊的 `misc/`、`temp/`、`new/` 等目录。
- 临时文件优先放到系统临时目录，不放入项目。
- 生成产物默认不进 git，除非它是项目交付物。

## 文档约定

`docs/` 下建议使用以下文件：

- `docs/requirements.md`：需求澄清结果。
- `docs/spec.md`：可验收的功能规格。
- `docs/design.md`：技术设计。
- `docs/plan.md`：实施计划和任务拆分。
- `docs/review.md`：阶段审查记录。

文档命名规则：

- 文件名使用英文小写和连字符，例如 `input-source-switching.md`。
- 文档正文优先中文。
- 决策记录写清楚：背景、选项、决定、后果。

## macOS 输入法约束

- 不默认修改系统设置。
- 不默认安装全局依赖。
- 不默认写入 `/Library`、`~/Library` 或系统级配置目录。
- 需要访问辅助功能、输入监听、自动化权限时，先在文档中说明原因、范围和替代方案。
- 涉及输入法 bundle id、快捷键事件、权限模型的判断，需要用本机验证或官方资料确认。

## 开发流程

每个功能按以下阶段推进：

1. 需求澄清：明确用户动作、系统状态、成功标准、失败场景。
2. Spec：写出可验收行为，不讨论实现细节。
3. 设计：比较技术路径，选最直接可维护的方案。
4. 计划：拆成可执行步骤，标明验证方式。
5. TDD：先写测试，再写实现，再重构。
6. 审查：先自查，再根据需要做第二阶段审查。
7. 完成：跑验证，汇报改动、验证结果、剩余风险。

## 验证约定

当前技术栈为 Swift Package Manager。当前本机只有 CommandLineTools，缺少 Swift `Testing` / `XCTest` 模块；因此第一版使用项目自带轻量测试 runner。实现阶段默认验证命令为：

- `env CLANG_MODULE_CACHE_PATH=.build/module-cache XDG_CACHE_HOME=.build/xdg-cache swift run --disable-sandbox --cache-path .build/swiftpm-cache rimedou-tests`

文档阶段仍需检查文档是否完整、互相一致。

## Git 约定

- 默认不创建 commit，除非用户要求。
- 默认不 push。
- 分支名如需创建，使用 `codex/` 前缀，除非用户指定其他命名。
- 工作区可能存在用户改动，不能擅自回退。

## 清理约定

- 不删除用户文件。
- 不删除未确认用途的文件。
- 清理构建产物前，先确认它确实是可再生成产物。
- 依赖、缓存、临时产物应通过 `.gitignore` 或技术栈工具管理。
